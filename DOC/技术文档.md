## 1. 技术选型

### 1.1 技术栈总览

```
┌──────────────────────────────────────────────────┐
│                    前端层                        │
│  HTML5 + CSS3 + Vanilla JS + Bootstrap 5.3       │
│  (无前端构建工具，直接运行)                       │
├──────────────────────────────────────────────────┤
│                   通信层                         │
│  HTTP RESTful API · JSON · CORS                 │
├──────────────────────────────────────────────────┤
│                   后端层                         │
│  Spring Boot 2.7.18                             │
│  MyBatis-Plus 3.5.5                             │
│  Apache POI 5.2.5 · Hutool 5.8.25               │
│  Lombok · Bean Validation                       │
├──────────────────────────────────────────────────┤
│                   数据层                         │
│  MySQL 8.0 · HikariCP 连接池                    │
└──────────────────────────────────────────────────┘
```

### 1.2 选型理由

| 技术 | 选型理由 |
|------|---------|
| **Spring Boot 2.7.18** | 当前最稳定的 2.x 版本，生态成熟；大量企业仍在使用，适合实习生学习 |
| **MyBatis-Plus** | 比 JPA 更灵活，比原生 MyBatis 更简洁；LambdaQueryWrapper 避免 SQL 字符串硬编码 |
| **Apache POI** | Java 生态 Excel 操作的事实标准，支持 .xlsx 格式，样式定制能力强 |
| **Bootstrap 5** | 最流行的 CSS 框架，CDN 引入零配置，组件丰富，响应式布局 |
| **原生 JavaScript** | 练手项目不宜引入 Vue/React 构建链；原生 JS 有助于理解 DOM 操作和 Fetch API |
| **HikariCP** | Spring Boot 2.x 默认连接池，性能业界领先 |

### 1.3 不选型的说明

| 未选用 | 原因 |
|--------|------|
| Spring Security / Shiro | v1.0 聚焦业务功能，暂不需要用户认证 |
| Redis | 数据量小，无需缓存 |
| Vue / React | 增加学习成本，练手项目保持简单 |
| Docker | 先让项目跑起来，容器化在后续版本引入 |

## 3. 后端架构

### 3.1 分层架构

```
┌─────────────────────────────────────────┐
│  Controller 层 (REST API)              │
│  · 接收 HTTP 请求                       │
│  · 参数校验 (@Valid / @Validated)       │
│  · 返回 Result<T> 统一响应              │
├─────────────────────────────────────────┤
│  Service 层 (业务逻辑)                  │
│  · 业务规则校验                         │
│  · 组装查询条件                         │
│  · 调用 Mapper + 工具类                 │
├─────────────────────────────────────────┤
│  Mapper 层 (数据访问)                   │
│  · 继承 BaseMapper<T>                   │
│  · 使用 LambdaQueryWrapper 构建查询     │
├─────────────────────────────────────────┤
│  Entity 层 (数据模型)                   │
│  · 映射数据库表                         │
│  · JSR-303 校验注解                     │
└─────────────────────────────────────────┘
```

### 3.2 核心组件说明

#### 3.2.1 统一响应体 — `Result<T>`

所有 API 返回如下 JSON 结构：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... },
  "timestamp": 1719715200000
}
```

**设计意图**：
- 前端无需判断 HTTP 状态码，统一解析 `code` 字段；
- `message` 用于用户提示；
- `timestamp` 便于日志追踪和问题排查。

#### 3.2.2 全局异常处理 — `GlobalExceptionHandler`

使用 `@RestControllerAdvice` 拦截所有 Controller 抛出的异常：

| 异常类型 | HTTP状态 | 场景 |
|---------|---------|------|
| `MethodArgumentNotValidException` | 400 | @Valid 校验失败 |
| `BindException` | 400 | 参数绑定失败 |
| `ConstraintViolationException` | 400 | @RequestParam 校验失败 |
| `RuntimeException` | 400 | 业务异常（如"客户不存在"） |
| `Exception` | 500 | 未预期的系统异常 |

**优点**：避免在 Controller 中写 try-catch，集中管理异常处理逻辑。

#### 3.2.3 跨域配置 — `CorsConfig`

前后端分离必然涉及跨域问题。开发阶段配置允许所有来源：

```java
config.addAllowedOriginPattern("*");
```

> ⚠️ **生产环境**务必改为具体域名，如 `https://crm.example.com`。

#### 3.2.4 分页插件 — `MyBatisPlusConfig`

配置 `PaginationInnerInterceptor`，自动为分页查询拼接 `LIMIT` 语句：

```java
pagination.setMaxLimit(500L);   // 单次最多500条
pagination.setOverflow(true);   // 超出页码范围返回第一页
```

### 3.3 数据流向

```
用户操作 → 前端 JS → HTTP 请求 → Controller
                                   ↓
                              参数校验
                                   ↓
                              Service 层
                              ├── 构建 LambdaQueryWrapper
                              ├── 调用 Mapper.selectPage()
                              └── 调用 ExcelUtil.export()
                                   ↓
                              MyBatis-Plus
                              ├── 生成 SQL
                              └── 通过 HikariCP 连接池执行
                                   ↓
                              MySQL 返回结果
                                   ↓
                              封装为 Result<T>
                                   ↓
                              JSON 序列化返回前端
```

## 4. 前端架构

### 4.1 页面路由

| 页面 | 文件 | 功能 |
|------|------|------|
| 主页面 | `index.html` | 客户列表、CRUD、筛选、导出 |
| 共享浏览 | `pages/share.html` | 已签约客户只读浏览 |

### 4.2 JS 模块划分

| 文件 | 职责 |
|------|------|
| `api.js` | 封装所有后端 API 调用，统一错误处理和超时控制 |
| `app.js` | 页面渲染、用户交互、表单校验、状态管理 |

### 4.3 前端状态管理

使用简单的全局对象 `STATE` 管理状态（本项目不用 Redux/Vuex）：

```javascript
const STATE = {
    currentPage: 1,      // 当前页码
    pageSize: 10,        // 每页条数
    filters: { ... },    // 筛选条件
    editingId: null,     // 编辑中的客户ID
    customerModal: null, // Bootstrap Modal 实例
    customerTypes: [],   // 客户类型枚举
};
```

### 4.4 前端安全措施

| 措施 | 实现方式 |
|------|---------|
| XSS 防护 | `escapeHtml()` 函数 — 使用 `textContent` 转义 HTML 特殊字符 |
| 表单校验 | 提交前检查必填字段和邮箱格式 |
| 删除确认 | 原生 `confirm()` 二次确认 |

## 5. 数据库设计

### 5.1 核心表结构

#### `customer` — 客户信息主表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| company_name | VARCHAR(200) | NOT NULL | 公司名称 |
| contact_person | VARCHAR(100) | NOT NULL | 联系人 |
| phone | VARCHAR(30) | | 电话 |
| email | VARCHAR(100) | | 邮箱 |
| country | VARCHAR(50) | | 国家 |
| address | VARCHAR(500) | | 地址 |
| website | VARCHAR(200) | | 网站 |
| wechat | VARCHAR(50) | | 微信号 |
| customer_type | VARCHAR(50) | NOT NULL | 客户类型 |
| service_needs | TEXT | | 服务需求 |
| source | VARCHAR(100) | | 来源渠道 |
| status | VARCHAR(20) | DEFAULT '潜在客户' | 跟进状态 |
| remark | TEXT | | 备注 |
| created_by | VARCHAR(100) | | 录入人 |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | ON UPDATE CURRENT_TIMESTAMP | 更新时间 |
| is_deleted | TINYINT(1) | DEFAULT 0 | 逻辑删除 |

**索引设计**：

```sql
INDEX idx_company_name (company_name)   -- 按公司名搜索
INDEX idx_customer_type (customer_type) -- 按类型筛选
INDEX idx_status (status)               -- 按状态筛选
INDEX idx_country (country)             -- 按国家筛选
INDEX idx_created_at (created_at)       -- 按时间排序
```

**设计考量**：
- 使用 `VARCHAR` 而非 `ENUM` 存储枚举值：便于后期扩展类型，避免 ALTER TABLE；
- `service_needs` 和 `remark` 使用 `TEXT` 类型：支持长文本；
- 逻辑删除字段 `is_deleted`：配合 MyBatis-Plus `@TableLogic` 自动过滤；
- `updated_at` 使用 `ON UPDATE CURRENT_TIMESTAMP`：数据库级别自动维护。

### 5.2 扩展表（预留）

#### `operation_log` — 操作日志表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| customer_id | BIGINT | 关联客户ID |
| operation_type | VARCHAR(50) | 操作类型 |
| content | TEXT | 操作描述 |
| operator | VARCHAR(100) | 操作人 |
| created_at | DATETIME | 操作时间 |

> 此表已在 SQL 脚本中建立，业务逻辑在 v1.1 版本对接。

## 6. 接口设计原则

| 原则 | 说明 |
|------|------|
| RESTful 风格 | URL 使用名词复数，HTTP 方法表达操作语义 |
| 统一前缀 | `/api/customers` |
| 统一响应 | `Result<T>` 封装 code、message、data |
| POST 查询 | 复杂查询使用 POST + JSON Body（非 GET + Query String） |
| 逻辑删除 | DELETE 请求执行逻辑删除，非物理删除 |

## 7. 部署指南

### 7.1 环境要求

| 组件 | 最低版本 | 推荐版本 |
|------|---------|---------|
| JDK | 8 | 11 或 17 |
| Maven | 3.6 | 3.8+ |
| MySQL | 8.0 | 8.0.33+ |
| 浏览器 | Chrome 90 | Chrome 120+ |

### 7.2 后端部署步骤

```bash
# 1. 初始化数据库
mysql -u root -p < sql/init.sql

# 2. 修改数据库密码
vim backend/src/main/resources/application.yml
# 修改 spring.datasource.password

# 3. 编译打包
cd backend
mvn clean package -DskipTests

# 4. 启动
java -jar target/crm-backend-1.0.0.jar

# 或使用 Maven 直接启动
mvn spring-boot:run
```

### 7.3 前端部署步骤

**方式一：直接打开（开发/演示）**

直接用浏览器打开 `frontend/index.html`。

> ⚠️ 如后端端口非 8080，需修改 `frontend/js/api.js` 中的 `BASE_URL`。

**方式二：Nginx 部署（生产环境）**

```nginx
server {
    listen 80;
    server_name crm.example.com;

    # 前端静态资源
    location / {
        root /var/www/crm/frontend;
        index index.html;
    }

    # 后端 API 反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### 7.4 配置检查清单

- [ ] 数据库是否已创建并导入初始数据
- [ ] `application.yml` 中数据库用户名密码是否正确
- [ ] 后端端口是否与前端 API 地址一致
- [ ] 生产环境是否修改了 CORS 配置为具体域名
- [ ] MySQL 时区配置是否与 `serverTimezone=Asia/Shanghai` 一致

## 8. 开发规范

### 8.1 Java 编码规范

- 类名：大驼峰（`CustomerController`）
- 方法名：小驼峰（`listByCondition`）
- 包名：全小写（`com.crm.controller`）
- Controller 方法尽量精简，业务逻辑下沉到 Service；
- 使用 Lombok 注解减少样板代码（`@Data`, `@Slf4j`, `@RequiredArgsConstructor`）；
- 日志使用 `@Slf4j` + `log.info/warn/error`，不使用 `System.out.println`。

### 8.2 前端编码规范

- 使用 IIFE 或模块模式避免全局变量污染（`const API = (() => { ... })()`）；
- DOM 操作前先检查元素是否存在；
- 异步请求使用 `async/await`，配合 try-catch 处理异常；
- HTML 输出必须经过 `escapeHtml()` 转义。

### 8.3 SQL 规范

- 表名和字段名使用小写蛇形命名（`company_name`）；
- 每张表必须有 `created_at` 和 `updated_at` 审计字段；
- 删除操作使用逻辑删除，严禁 `DROP` 和物理 `DELETE`；
- 索引命名：`idx_字段名`。
