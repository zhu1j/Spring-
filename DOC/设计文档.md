## 1. 总体设计

### 1.1 设计目标

- **简单实用**：实习生可独立理解和维护；
- **企业级规范**：代码按照企业项目标准分层、命名、注释；
- **前后端分离**：后端纯 API，前端纯静态页面，通过 HTTP JSON 通信；
- **可扩展**：为后续功能（用户认证、跟进记录、数据看板）预留扩展点。

### 1.3 设计模式应用

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| 分层架构 | 整体 | Controller → Service → Mapper 三层分离 |
| 静态工厂方法 | `Result.java` | `Result.ok()`, `Result.fail()` 统一创建响应对象 |
| 建造者模式 | `Result.java`, `Customer.java` | Lombok `@Builder` 注解 |
| 模板方法 | `BaseMapper<T>` | MyBatis-Plus 提供通用 CRUD 模板 |
| 单例（作用域） | Spring Bean | `@Service`, `@Controller` 默认单例 |
| 前端模块模式 | `api.js` | IIFE 封装，暴露统一 API 接口 |

## 2. 模块设计

### 2.1 模块划分

```
CRM 系统
├── 客户录入模块        create / update
├── 客户查询模块        list / getById
├── 客户删除模块        delete (逻辑删除)
├── 数据导出模块        export (Excel)
├── 共享浏览模块        share (只读列表)
└── 枚举查询模块        types (下拉选项)
```

### 2.2 模块间依赖

```
客户录入 ──→ CustomerMapper ──→ MySQL
客户查询 ──→ CustomerMapper ──→ MySQL
客户删除 ──→ CustomerMapper ──→ MySQL
数据导出 ──→ CustomerMapper ──→ ExcelUtil ──→ HTTP Response
共享浏览 ──→ CustomerService.listForShare() ──→ CustomerMapper
枚举查询 ──→ CustomerService.getCustomerTypes() (内存常量)
```

各模块通过 `CustomerService` 接口解耦，Controller 只依赖 Service 接口，不直接依赖 Mapper。

### 3.2 索引设计说明

| 索引名 | 字段 | 类型 | 使用场景 |
|--------|------|------|---------|
| PRIMARY | id | 聚簇索引 | 主键查询 |
| idx_company_name | company_name | 普通索引 | 按公司名模糊搜索 |
| idx_customer_type | customer_type | 普通索引 | 按类型筛选列表 |
| idx_status | status | 普通索引 | 按状态筛选、共享浏览只查"已签约" |
| idx_country | country | 普通索引 | 按国家筛选 |
| idx_created_at | created_at | 普通索引 | 默认排序字段 |

**索引设计原则**：
- 在 WHERE 条件中频繁出现的字段建索引；
- 选择性高的字段优先（如 status 只有5个值，但与 customer_type 组合后选择性提高）；
- 避免过多索引影响写入性能（当前数据量小，6个索引可接受）。

### 3.3 字符集选择

- 数据库字符集：`utf8mb4`（支持 emoji 和特殊中文）
- 排序规则：`utf8mb4_unicode_ci`（不区分大小写，适合多语言搜索）

## 4. 接口设计

### 4.1 RESTful API 设计

| 方法 | 路径 | 请求体 | 响应 | 说明 |
|------|------|--------|------|------|
| POST | `/api/customers` | Customer JSON | `Result<Customer>` | 新增 |
| PUT | `/api/customers/{id}` | Customer JSON | `Result<Customer>` | 全量更新 |
| DELETE | `/api/customers/{id}` | — | `Result<Void>` | 逻辑删除 |
| GET | `/api/customers/{id}` | — | `Result<Customer>` | 查询详情 |
| POST | `/api/customers/list` | CustomerQueryDTO | `Result<PageData>` | 分页查询 |
| GET | `/api/customers/export` | Query Params | 文件流(.xlsx) | 导出 |
| POST | `/api/customers/share` | CustomerQueryDTO | `Result<PageData>` | 共享浏览 |
| GET | `/api/customers/types` | — | `Result<List<String>>` | 客户类型枚举 |

### 4.2 为什么"列表查询"用 POST 而非 GET？

**原因**：
1. GET 请求的 Query String 长度有限制（浏览器 ≈ 2000 字符，虽然 HTTP 协议无限制）；
2. 复杂 JSON 查询条件（嵌套对象/数组）无法在 Query String 中优雅表达；
3. 本项目查询条件尚简单，但为了保持接口风格一致和后续扩展，统一使用 POST + JSON Body。

> 注：导出接口因需要支持浏览器直接访问下载，使用了 GET + Query Params。

### 4.3 分页响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "records": [ ... ],
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "pages": 10
  }
}
```

## 6. 关键流程设计

### 6.1 新增客户流程

```
用户点击"新增客户"
    │
    ▼
弹出模态框（空白表单）
    │
    ▼
用户填写信息 → 点击"保存"
    │
    ▼
前端校验必填字段 ──失败──→ 提示错误信息，留在表单
    │ 成功
    ▼
POST /api/customers (JSON Body)
    │
    ▼
后端 @Valid 校验 ──失败──→ 全局异常处理 → 返回400 + 错误信息
    │ 成功
    ▼
CustomerService.create()
    │
    ├─ 设置默认状态"潜在客户"
    ├─ CustomerMapper.insert()
    └─ 返回带自增ID的Customer对象
    │
    ▼
前端收到 Result{code:200}
    │
    ├─ 关闭模态框
    ├─ 显示 Toast "客户录入成功"
    └─ 刷新列表（跳转到第1页）
```

### 6.2 Excel 导出流程

```
用户点击"导出Excel"
    │
    ▼
前端收集当前筛选条件
    │
    ▼
window.open(/api/customers/export?customerType=xxx&status=xxx&...)
    │
    ▼
CustomerController.export()
    │
    ├─ 构建 CustomerQueryDTO
    ├─ 调用 CustomerService.exportExcel()
    │      │
    │      ├─ 构建 LambdaQueryWrapper（不设分页）
    │      ├─ CustomerMapper.selectList() → 获取全部数据
    │      └─ ExcelUtil.export(response, data, "客户资料")
    │             │
    │             ├─ 创建 XSSFWorkbook
    │             ├─ 写入表头（加粗、蓝色背景）
    │             ├─ 逐行写入数据
    │             ├─ 设置列宽
    │             ├─ 冻结首行
    │             └─ response.getOutputStream() 输出文件流
    │
    ▼
浏览器识别 Content-Disposition: attachment
    │
    ▼
触发文件下载（客户资料.xlsx）
```

### 6.3 共享浏览流程

```
外部用户打开 share.html
    │
    ▼
页面加载 → 自动调用 POST /api/customers/share
    │
    ▼
CustomerService.listForShare()
    │
    ├─ 构建 LambdaQueryWrapper
    ├─ 强制添加条件: eq(status, "已签约")
    ├─ 分页查询
    └─ 返回 Page<Customer>
    │
    ▼
前端渲染表格（仅显示公司名、联系人、国家、类型、时间）
    │
    ▼
注：敏感字段（电话、邮箱、微信、地址）只在后端返回，
但前端 share.html 不渲染这些字段，实现前端层面的脱敏展示。
```

## 7. 异常处理设计

### 7.1 异常分类

```
异常体系
├── 参数校验异常 (400)
│   ├── MethodArgumentNotValidException (@Valid 失败)
│   ├── BindException (参数绑定失败)
│   └── ConstraintViolationException (约束违反)
├── 业务异常 (400)
│   └── RuntimeException (如"客户不存在")
└── 系统异常 (500)
    └── Exception (未预期的异常)
```

### 7.2 异常处理策略

| 策略 | 说明 |
|------|------|
| 统一拦截 | `@RestControllerAdvice` 全局拦截，避免 Controller 中散落 try-catch |
| 区分状态码 | 400（客户端错误）vs 500（服务端错误） |
| 日志分级 | 校验失败 → WARN，业务异常 → ERROR（含堆栈），系统异常 → ERROR |
| 信息脱敏 | 生产环境不返回 `e.getMessage()` 给前端，避免泄露内部细节 |

## 8. 扩展性设计

### 8.1 预留扩展点

| 扩展项 | 预留方式 | 说明 |
|--------|---------|------|
| 用户认证 | Controller 层增加拦截器/过滤器 | 不影响 Service 层 |
| 权限管理 | Service 层增加 `userId` 参数 | 数据按用户隔离 |
| 多租户 | SQL 增加 `tenant_id` 字段 | MyBatis-Plus 支持多租户插件 |
| 跟进记录 | 新增 `follow_up` 表，关联 `customer_id` | 独立模块，不影响现有功能 |
| 数据看板 | 新增统计 Service + 图表前端 | 读取现有数据，不修改表结构 |
| 导入功能 | 新增 `ExcelUtil.importExcel()` | 复用 POI 依赖 |
| 移动端 | API 不变，新增前端项目 | 后端无需改动 |
| 消息队列 | Service 层事件发布 | 客户创建/状态变更时触发通知 |

### 8.2 技术债务记录

| 项 | 说明 | 优先级 |
|----|------|--------|
| 缺少单元测试 | 当前仅有空测试类，需补充 Service 层测试 | 高 |
| 缺少接口文档自动生成 | 可引入 Swagger/Knife4j | 中 |
| 前端无构建工具 | 可引入 Vite + Vue3 重构 | 低 |
| 缺少 CI/CD 配置 | 可引入 GitHub Actions / Jenkins | 中 |
| 硬编码枚举值 | 可改为数据库字典表 | 低 |
